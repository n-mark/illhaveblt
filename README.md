## Клиент
### 1. Первый визит и определение местоположения

```gherkin
Функционал: Определение местоположения при первом визите

  Как новый посетитель сайта
  Пользователь хочет чтобы система определила его местоположение
  Чтобы сразу показали актуальные акции и адрес пользователя был заполнен автоматически

  Сценарий: Первый визит — пользователь даёт разрешение на геолокацию
    Когда пользователь открывает сайт впервые
    Тогда пользователь видит каталог продукции
    И система запрашивает разрешение на определение местоположения пользователя
    Когда пользователь нажимает "Разрешить"
    Тогда система определяет координаты пользователя
    И автоматически выставляет адрес доставки согласно определённому местоположению

  Сценарий: Первый визит — пользователь отказывает в геолокации
    Когда пользователь открывает сайт впервые
    Тогда пользователь видит каталог продукции
    И система запрашивает разрешение на определение местоположения пользователя
    Когда пользователь нажимает "Запретить"
    Тогда система не определяет местоположение пользователя
    И адрес доставки остаётся незаполненным
    И акции, зависящие от региона, не отображаются

  Сценарий: Первый визит — пользователь самостоятельно указывает местоположение
    Когда пользователь открывает сайт впервые
    Тогда пользователь видит каталог продукции
    И система запрашивает разрешение на определение местоположения пользователя
    Когда пользователь нажимает "Запретить"
    Тогда система не определяет местоположение пользователя
    Но если пользователь выбрал местоположение самостоятельно через всплывающее окно
    Тогда система выставляет адрес согласно выбранному местоположению

  Сценарий: Повторный визит с того же устройства — геолокация не запрашивается
    Дано: пользователь уже посещал сайт с этого устройства
    И системе известен адрес пользователя
    Когда пользователь открывает сайт
    Тогда система не запрашивает разрешение на геолокацию повторно
    И не просит указать адрес пользователя
    И сразу отображает каталог с актуальными акциями для региона пользователя
```

### 2. Авторизация существующего пользователя

```gherkin
Функционал: Вход в существующий аккаунт

  Как зарегистрированный пользователь
  Пользователь хочет войти в свой аккаунт
  Чтобы не вводить данные заново при каждом заказе

  Сценарий: Успешный вход через кнопку "Войти"
    Дано: пользователь находится на главной странице с каталогом
    Когда пользователь нажимает кнопку "Войти" в правом верхнем углу
    И вводит свой номер телефона
    И вводит последние цифры входящего звонка для подтверждения
    Тогда система успешно авторизует пользователя
    И в правом верхнем углу отображается имя пользователя вместо кнопки "Войти"
    И система не запрашивает никаких дополнительных данных при добавлении товаров в заказ
```

### 3. Отображение акций

```gherkin
Функционал: Отображение акций в зависимости от региона пользователя

  Как покупатель
  Пользователь хочет видеть актуальные акции для своего региона
  Чтобы воспользоваться выгодными предложениями

  Сценарий: Отображение региональных акций при известном местоположении
    Дано: в системе опредено местоположение пользователя
    Когда пользователь просматривает каталог
    Тогда над списком позиций отображается блок с доступными акциями для страны и региона пользователя

  Сценарий: Отображение акции рядом с конкретной позицией
    Дано: в системе опредено местоположение пользователя
    И в регионе пользователя действует акция на позицию "Клаб-сэндвич"
    Когда пользователь просматривает список позиций
    Тогда рядом с карточкой "Клаб-сэндвич" отображается значок или плашка с акцией

  Сценарий: Акции не отображаются при отстутствии местоположения
    Дано: система не знает местоположение пользователя
    Когда пользователь просматривает каталог
    Тогда блок с региональными акциями над списком не отображается
    И плашки акций рядом с позициями не отображаются
```

### 4. Просмотр и выбор позиции

```gherkin
Функционал: Просмотр карточки позиции из каталога

  Как покупатель
  Пользователь хочет узнать подробности о позиции перед добавлением в заказ
  Чтобы сделать осознанный выбор с учётом состава и пищевой ценности

  Сценарий: Открытие карточки позиции
    Когда пользователь нажимает на карточку позиции "БЛТ-сэндвич" в каталоге
    Тогда открывается страница с подробным описанием позиции
    И отображаются ингредиенты в составе
    И отображается таблица БЖУ (белки, жиры, углеводы)

  Сценарий: Позиция поддерживает выбор опций приготовления
    Дано: позиция "Клаб-сэндвич" позволяет выбирать опции
    Когда пользователь открывает карточку "Клаб-сэндвич"
    Тогда под описанием отображается список доступных опций приготовления
    И пользователь может выбрать нужные опции перед добавлением в заказ

  Сценарий: Позиция не поддерживает выбор опций приготовления
    Дано: позиция "Кукис" не имеет вариаций
    Когда пользователь открывает карточку "Кукис"
    Тогда список опций приготовления не отображается
    И доступна только кнопка "Добавить в заказ"
```

### 5. Добавление позиции в заказ

```gherkin
Функционал: Добавление позиции в заказ

  Как покупатель
  Пользователь хочет добавить выбранную позицию в заказ
  Чтобы перейти к его оформлению

  Сценарий: Добавление товара при уже разрешённой геолокации
    Дано: пользователь уже дал разрешение на определение местоположения
    И местоположение успешно определено
    Когда пользователь нажимает кнопку "Добавить в заказ" на карточке позиции
    Тогда система добавляет товар в заказ без дополнительных запросов
    И в корзине отображается добавленная позиция

  Сценарий: Добавление товара при отсутствии разрешения на геолокацию
    Дано: пользователь не давал разрешения на определение местоположения
    И пользователь не вошёл в свой аккаунт
    Когда пользователь нажимает кнопку "Добавить в заказ" на карточке позиции
    Тогда система отображает всплывающее окно с запросом адреса
    И в окне есть строка ввода адреса
    И в окне отображается интерактивная карта
    Когда пользователь вводит адрес и подтверждает его
    Тогда система добавляет товар в заказ
    И адрес сохраняется как адрес доставки

  Сценарий: Добавление товара авторизованным пользователем
    Дано: пользователь уже вошёл в свой аккаунт
    Когда пользователь нажимает кнопку "Добавить в заказ"
    Тогда система добавляет товар без запроса местоположения или данных авторизации
```

### 6. Переход к оформлению — авторизация через телефон

```gherkin
Функционал: Авторизация пользователя при переходе к оформлению заказа

  Как неавторизованный покупатель
  Пользователь хочет войти или зарегистрироваться при оформлении заказа
  Чтобы завершить покупку

  Сценарий: Переход к оформлению неавторизованным пользователем
    Дано: пользователь добавил позиции в заказ
    И пользователь не авторизован в системе
    Когда пользователь нажимает кнопку "Далее"
    Тогда система отображает всплывающее окно с запросом номера телефона
    И поясняет что номер нужен для входа или регистрации

  Сценарий: Подтверждение номера телефона через входящий звонок
    Дано: пользователь ввёл свой номер телефона в форму авторизации
    Когда пользователь нажимает "Получить код"
    Тогда система инициирует входящий звонок на указанный номер
    И отображает поле для ввода последних цифр номера входящего звонка
    Когда пользователь вводит последние цифры правильно
    Тогда номер телефона считается подтверждённым

  Сценарий: Неверный код подтверждения
    Дано: система инициировала входящий звонок
    Когда пользователь вводит неверные последние цифры номера
    Тогда система отображает сообщение "Неверный код. Попробуйте ещё раз"
    И поле ввода остаётся доступным для повторного ввода

  Сценарий: Переход к оформлению уже авторизованным пользователем
    Дано: пользователь уже вошёл в свой аккаунт
    Когда пользователь нажимает кнопку "Далее"
    Тогда система не отображает никаких всплывающих окон
    И сразу переходит к экрану оформления заказа
```

### 7. Регистрация нового пользователя

```gherkin
Функционал: Регистрация нового пользователя после подтверждения телефона

  Как новый пользователь
  Пользователь хочет создать аккаунт через номер телефона
  Чтобы сохранить свои данные и оформить заказ

  Сценарий: Создание аккаунта когда телефон не зарегистрирован
    Дано: пользователь успешно подтвердил свой номер телефона
    И к данному номеру не привязан ни один аккаунт
    Тогда система уведомляет пользователя "Аккаунт с таким номером не найден"
    И предлагает создать новый аккаунт
    Когда пользователь вводит своё имя и фамилию
    И ставит галочку "Принимаю пользовательское соглашение"
    И нажимает "Создать аккаунт"
    Тогда система создаёт аккаунт пользователя
    И автоматически авторизует пользователя
    И переходит к экрану оформления заказа

  Сценарий: Попытка создать аккаунт без принятия пользовательского соглашения
    Дано: система предложила пользователю создать аккаунт
    Когда пользователь заполняет имя и фамилию
    И не принимает пользовательское соглашение
    И нажимает "Создать аккаунт"
    Тогда система не создаёт аккаунт
    И отображает предупреждение "Необходимо принять пользовательское соглашение"

  Сценарий: Телефон уже привязан к аккаунту — вход без регистрации
    Дано: пользователь успешно подтвердил свой номер телефона
    И к данному номеру уже привязан аккаунт
    Тогда система автоматически авторизует пользователя без дополнительных шагов
    И сразу переходит к экрану оформления заказа
```

### 8. Оформление заказа — доставка

```gherkin
Функционал: Оформление заказа с доставкой

  Как авторизованный покупатель
  Пользователь хочет оформить доставку заказа до своего адреса
  Чтобы получить сэндвич не выходя из дома или офиса

  Контекст:
    Дано: пользователь авторизован в системе
    И пользователь находится на экране оформления заказа

  Сценарий: Выбор доставки с ранее определённым адресом
    Когда пользователь выбирает способ получения "Доставка"
    Тогда система отображает список выбранных позиций заказа
    И отображает адрес доставки, определённый ранее по геолокации
    И показывает примерное время доставки

  Сценарий: Уточнение адреса доставки
    Дано: пользователь выбрал способ получения "Доставка"
    И система отобразила адрес доставки по геолокации
    Когда пользователь нажимает на поле адреса и вносит уточнения (квартира, подъезд, этаж)
    Тогда система сохраняет уточнённый адрес
    И пересчитывает примерное время доставки если оно изменилось

  Сценарий: Выбор способа оплаты при доставке
    Дано: пользователь выбрал способ получения "Доставка"
    Когда пользователь выбирает способ оплаты "На сайте"
    Тогда система переводит пользователя на страницу ввода платёжных данных
    Когда пользователь выбирает способ оплаты "При получении"
    Тогда система подтверждает заказ без перехода к оплате онлайн

  Сценарий: Подтверждение заказа с доставкой
    Дано: пользователь выбрал доставку и способ оплаты
    Когда пользователь нажимает "Подтвердить заказ"
    Тогда система создаёт заказ и присваивает ему номер
    И пользователь получает уведомление с деталями заказа и ожидаемым временем доставки
```

### 9. Оформление заказа — самовывоз

```gherkin
Функционал: Оформление заказа с самовывозом

  Как авторизованный покупатель
  Пользователь хочет забрать готовый заказ из ближайшего ресторана самостоятельно
  Чтобы сэкономить на доставке и получить заказ быстрее

  Контекст:
    Дано: пользователь авторизован в системе
    И пользователь находится на экране оформления заказа
    И система знает местоположение пользователя

  Сценарий: Выбор самовывоза — отображение ближайшего ресторана
    Когда пользователь выбирает способ получения "Самовывоз"
    Тогда система определяет ближайший к местоположению пользователя ресторан
    И отображает название и адрес ближайшего ресторана
    И показывает приблизительное время к которому следует прибыть за заказом

  Сценарий: Построение маршрута пешком
    Дано: пользователь выбрал самовывоз
    Когда пользователь выбирает способ перемещения "Пешком"
    Тогда система запрашивает маршрут у внешнего картографического сервиса
    И отображает карту с оптимальным пешим маршрутом от местоположения пользователя до ресторана
    И показывает расчётное время в пути пешком

  Сценарий: Построение маршрута на личном автомобиле
    Дано: пользователь выбрал самовывоз
    Когда пользователь выбирает способ перемещения "Личное авто"
    Тогда система запрашивает маршрут у внешнего картографического сервиса с учётом трафика
    И отображает карту с оптимальным маршрутом для авто
    И показывает расчётное время в пути с учётом пробок
```

## 1. Order API
**Service Name:** Order API
<br>
**Description:** Управление жизненным циклом заказа: создание, изменение, отмена, отслеживание статусов.

### Consumer Tasks
- Клиент создаёт/меняет/отменяет заказ.
- Работник ресторана обновляет статус заказа (начало/завершение готовки).

### Dependencies
**Service Dependencies** (синхронные REST)
- Payment API: `POST /internal/authorize`, `POST /internal/capture`, `POST /internal/refund`
- Delivery API: запрос доставки, оповещение о готовности - асинхронно, отмена - `POST /internal/cancelDelivery`
- Notification API: асинхронно

**Event Subscriptions** (Kafka)
- `delivery.events`: `CourierAssigned`, `CourierPickedUp`, `CourierDelivered`, `DeliveryFailed`, `CustomerNotReachable`

**Logic/Rules**
- Нельзя отменить заказ после статуса `Cooking`.
- При самовывозе не вызывать Delivery API.

### Interface
**Queries** (REST GET)
- `GET /api/v1/orders/{orderId}` – детали заказа
- `GET /api/v1/orders/{orderId}/status` – текущий статус
- `GET /api/v1/orders?userId={userId}&limit=10&offset=0` – список заказов пользователя
- `GET /api/v1/orders/{orderId}/eta` – расчётное время доставки

**Commands** (REST POST/PUT/DELETE)

- `POST /api/v1/orders` – создать заказ
  *Body:* `{ userId, restaurantId, items, pickupType, address, paymentMethodId }`
  *Response:* `{ orderId, status, total, estimatedPickupTime }`

- `PUT /api/v1/orders/{orderId}` – изменить заказ
  *Body:* `{ items: [...] }`

- `POST /api/v1/orders/{orderId}` – отменить заказ
  *Body:* `{reason, comment}`

- `POST /api/v1/orders/{orderId}/promo` – применить промокод
  *Body:* `{ code }` → возвращает скидку

- `PUT /api/v1/orders/{orderId}/status`
  *Body:* `{ status }`



**Events Published** (Kafka)
- `OrderCreated`, `OrderRevised`, `OrderCancelled`, `OrderCookingStarted`, `OrderReadyForPickup`, `OrderPickedUp`, `OrderDelivered`, `OrderFailed`

## 2. Delivery API
**Service Name:** Delivery API
<br>
**Description:** Управление курьерами, назначение заказов, отслеживание геопозиции в реальном времени, обработка проблем доставки.

### Consumer Tasks
- Курьер получает заказы на доставку, принимает/отклоняет заказ, отмечает получение и доставку.

### Dependencies
**Service Dependencies**
- Картографический сервис (REST: `GET /routes`, `GET /distance`)

**Event Subscriptions** (Kafka)
- `order.events`: `OrderNeedsDelivery`,`OrderReadyForPickup`

**Logic/Rules**
- Курьер может принять не более 3 активных заказов одновременно.
- При отклонении заказ переназначается следующему ближайшему курьеру.
- Если клиент не вышел на связь 10 минут – доставка переводится в статус DeliveryFailed.

### Interface
**Queries** (REST GET)
- `GET /api/v1/deliveries/{orderId}/courier-location` – текущая геопозиция курьера
  *Response:* `{ lat, lon, lastUpdate }`
- `GET /api/v1/deliveries/{orderId}/eta` – расчётное время прибытия
- `GET /api/v1/couriers/available?lat={lat}&lon={lon}&radius={radius}` – список свободных курьеров
- `GET /api/v1/couriers/{courierId}/history?from=2025-01-01&to=2025-01-31` – история доставок курьера

**Commands** (REST POST/PUT)

- `POST /api/v1/deliveries/{orderId}/accept` – курьер принимает заказ
  *Body:* `{ courierId }`

- `POST /api/v1/deliveries/{orderId}/reject` – отклонить
  *Body:* `{ courierId, reason }`

- `PUT /api/v1/deliveries/{orderId}/pickup` – курьер забрал заказ
  *Body:* `{ courierId }`

- `PUT /api/v1/deliveries/{orderId}/deliver` – доставлен
  *Body:* `{ courierId, paymentReceived (bool) }`

- `POST /api/v1/deliveries/{orderId}/problem` – сообщить о проблеме
  *Body:* `{ problemType, description }`

**Events Published** (Kafka)
- `CourierAssigned`, `CourierPickedUp`, `CourierDelivered`, `DeliveryFailed`, `CustomerNotReachable`

## 4. Management API
**Service Name:** Management API

**Description:** Управление ресторанами, меню (глобальным/региональным/локальным), акциями, сотрудниками.

### Consumer Tasks
- Региональный менеджер добавляет ресторан, назначает менеджеров.
- Локальный менеджер управляет доступностью позиций, акциями, часами работы.

### Dependencies
- Restaurant API: *cоздать ресторан:* `POST /internal/restaurant`, *настроить ресторан:* `PUT /internal/restaurant`, *создать меню:* `POST /internal/menu`,  *изменить меню:* `PUT /internal/menu/{id}`, *создать акцию:* `POST /internal/promo`, *изменить акцию:* `PUT /internal/promo/{id}`
- User API: *добавить сотрудника:* `POST /internal/user`, *изменить запись сотрудника (в т.ч. деактивировать):* `PUT /internal/user/{id}`, *изменить роль сотрудника:* `PUT /internal/user/{id}/role`, *закрепить ресторан(ы) за сотрудником* 'POST /internal/user/{id}/restaurants'
- Notification API *оповестить зарегистрированных пользователей региона о новой акции / промо:* `асинхронно`

**Logic/Rules**
- Локальный менеджер может только включать/отключать доступность позиции (не может менять цену/состав).
- Региональный менеджер управляет региональным меню и акциями.
- Глобальный менеджер создаёт глобальные позиции и акции, которые нельзя отключить локально.

### Interface
**Commands** (REST POST/PUT/DELETE)

- `POST /api/v1/restaurants` – создать ресторан
  *Body:* `{ name, address, regionId, franchiseOwnerId }`

- `PUT /api/v1/restaurants/{restaurantId}/status` – изменить статус (активен/закрыт)
  *Body:* `{ status }`

- `POST /api/v1/restaurants/{restaurantId}/menu/items` – добавить позицию в меню
  *Body:* `{ name, price, ingredients, options }`

- `PUT /api/v1/menu/items/{itemId}/availability` – включить/отключить доступность
  *Body:* `{ available: true/false }`

- `POST /api/v1/promotions` – создать акцию
  *Body:* `{ name, type, discount, startDate, endDate, applicableItems, applicableRestaurants }`

- `POST /api/v1/managers/local` – назначить локального менеджера
  *Body:* `{ userId, restaurantIds[] }`

**Events Published** (Kafka)
- `NewPromoCreated`, `PromoUpdated`

## 5. Analytics API
**Service Name:** Analytics API
<br>
**Description:** Агрегирует данные из событий всех сервисов для построения отчётов для менеджеров (локальных, региональных, глобальных) и владельцев франшизы.

### Consumer Tasks
- Локальный менеджер смотрит выручку и загрузку своих ресторанов.
- Региональный менеджер сравнивает рестораны в регионе.

### Dependencies
**Service Dependencies**
- Нет внешних вызовов

**Event Subscriptions** (Kafka)

## 9. Message API
**Service Name:** Message API
<br>

**Description:** Обмен сообщениями в реальном времени между клиентом, курьером, персоналом ресторана и поддержкой. WebSocket для мгновенной доставки, хранение истории переписки. Поддерживает заказные чаты и внутренние каналы ресторана.

### Consumer Tasks
- Клиент пишет курьеру во время доставки (например, уточнить адрес).
- Курьер отвечает клиенту или запрашивает помощь.
- Поддержка ресторана переписывается с клиентом по вопросам заказа.
- Управляющий ресторана отправляет объявления и задачи сотрудникам (повара, кассиры).
- Сотрудники ресторана общаются в общем канале смены.
### Dependencies
**Service Dependencies**
- Order API: `GET /internal/orders/{orderId}/participants` — получить участников заказа (клиент, курьер, ресторан)
- User API: `GET /internal/users/{userId}` — имя и аватар для отображения
- Restaurant API: `GET /internal/restaurants/{restaurantId}/staff` — список сотрудников ресторана с ролями

**Event Subscriptions** (Kafka)
- `order.events`: `OrderCreated` → создать комнату чата заказа
- `order.events`: `OrderDelivered`, `OrderCancelled` → закрыть комнату чата заказа (архивация)
- `delivery.events`: `CourierAssigned` → добавить курьера в комнату заказа
- `restaurant.events`: `StaffShiftStarted` → добавить сотрудника в канал смены
- `restaurant.events`: `StaffShiftEnded` → убрать сотрудника из канала смены


**Logic/Rules**
- **Заказные чаты (type=order):**
  - Комната создаётся автоматически при создании заказа.
  - Участники: клиент, курьер (после назначения), поддержка ресторана.
  - После доставки/отмены комната закрывается, новые сообщения невозможны.
  - Только участники заказа могут читать и писать.
- **Внутренние каналы (type=channel):**
  - Типы каналов:
    - `shift` — автоматический канал текущей смены ресторана (все сотрудники на смене).
    - `announce` — канал объявлений (только управляющий пишет, сотрудники читают).
    - `direct` — личная переписка между двумя сотрудниками (управляющий ↔ сотрудник, сотрудник ↔ сотрудник).
  - Канал `announce` создаётся один на ресторан, управляющий — единственный отправитель.
  - Канал `shift` создаётся автоматически, участники добавляются/удаляются по событиям смены.
  - Канал `direct` создаётся по запросу любого сотрудника ресторана.
  - Управляющий может писать во все каналы своего ресторана.
  - Сотрудник может писать только в `shift` и `direct`.
- **Общие правила:**
  - Сообщения сохраняются 30 дней после закрытия комнаты (order) или 90 дней (channel), затем удаляются.
  - Максимальная длина сообщения: 1000 символов.
  - Медиафайлы (фото) загружаются через Storage API, в сообщении передаётся URL.
  - Только участники заказа/канала могут читать и писать.

### Interface
**Queries** (REST GET)
- `GET /api/v1/chats/{roomId}/messages?limit=50&before={messageId}` — история сообщений (пагинация назад)
- `GET /api/v1/chats/by-order?orderId={orderId}` — получить комнату по заказу
- `GET /api/v1/chats?userId={userId}&type={order|channel}&limit=20` — список комнат пользователя с последним сообщением
- `GET /api/v1/chats/{roomId}/participants` — участники комнаты и их онлайн-статус
- `GET /api/v1/chats/channels?restaurantId={restaurantId}` — список внутренних каналов ресторана
**Commands** (REST POST / WebSocket)
- `POST /api/v1/chats/{roomId}/messages` — отправить сообщение (REST, fallback)
*Body:* `{ senderId, text, mediaUrl? }`
*Response:* `{ messageId, createdAt }`
- `POST /api/v1/chats/channels` — создать внутренний канал
*Body:* `{ type, restaurantId, name, participantIds? }`
*Response:* `{ roomId, type }`
- `POST /api/v1/chats/channels/direct` — начать личную переписку
*Body:* `{ restaurantId, fromUserId, toUserId }`
*Response:* `{ roomId }`
- `POST /api/v1/chats/channels/{roomId}/announce` — отправить объявление (только управляющий)
*Body:* `{ senderId, text, mediaUrl? }`
*Response:* `{ messageId, deliveryCount }`
- `PUT /api/v1/chats/{roomId}/read` — отметить сообщения прочитанными
*Body:* `{ userId, lastReadMessageId }`
- `POST /api/v1/chats/{roomId}/participants` — добавить участника в канал
*Body:* `{ userId }` (только для каналов типа channel, только админ)
- `DELETE /api/v1/chats/{roomId}/participants/{userId}` — удалить участника
- WebSocket events (клиент → сервер):
  - `message.send` `{ roomId, text, mediaUrl? }` — отправить сообщение
  - `message.typing` `{ roomId, isTyping }` — индикатор набора текста
  - `message.read` `{ roomId, lastReadMessageId }` — подтверждение прочтения
- WebSocket events (сервер → клиент):
  - `message.new` `{ messageId, roomId, senderId, text, mediaUrl, createdAt }` — новое сообщение
  - `message.typing` `{ roomId, userId, isTyping }` — кто-то печатает
  - `message.read` `{ roomId, userId, lastReadMessageId }` — кто-то прочитал
  - `channel.joined` `{ roomId, userId }` — сотрудник добавлен в канал
  - `channel.removed` `{ roomId, userId }` — сотрудник удалён из канала
  - `chat.closed` `{ roomId, reason }` — комната закрыта

**Events Published** (Kafka)
- `MessageSent`, `ChatRoomCreated`, `ChatRoomClosed`, `AnnouncementSent`, `ChannelCreated`
